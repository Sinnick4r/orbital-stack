# ADR-010: v0.5.0 scope - Space-Track and full skyfield propagation deferred to v0.6.0

## Status

Accepted - 2026-04-24

## Context

PLAN.md originally framed Fase 2 (v0.5.0) as "Canonical & Enrichment"
with three upstream sources (UNOOSA, Celestrak, Space-Track), cross-
source matching, and a skyfield-backed orbit_gap analysis. Estimated
effort: 25-35 hours.

Reviewing that scope against what the canonical dataset actually needs
to ship in v0.5.0 revealed two sub-scopes that add real complexity
without being required for the release to be valuable:

1. **Space-Track ingestion.** Space-Track provides historical TLEs
   and a more conservative government-maintained catalog. For the
   canonical schema's matching logic, however, Celestrak GP is
   sufficient - it exposes `INTLDES`, `OBJECT_NAME`, and
   `LAUNCH_DATE`, which is everything the hierarchical matcher
   needs. Space-Track's value is historical orbit tracking, not
   identity resolution.

2. **Full skyfield propagation.** Propagating TLEs over time to
   detect "registered but no recent propagation" events is the
   headline orbit_gap analysis, but it depends on Space-Track's
   historical feed to be meaningful. Without historical TLEs,
   propagation at a single snapshot is a point observation, not a
   gap analysis.

Meanwhile, both sub-scopes carry real costs:

- Space-Track authentication is session-based (login page + cookie),
  aggressively rate-limited (300 requests per hour on a standard
  account), and has terms of use that restrict redistribution - which
  directly affects what can be published on Hugging Face and Kaggle.
- skyfield's full dependency graph (numpy, sgp4, jplephem, plus
  ephemeris files) is ~50 MB of environment weight for a feature
  that, at v0.5.0 scope, is not used.

The question is whether to stretch v0.5.0 to cover both, or land a
narrower v0.5.0 sooner and move the deferred work into a named
follow-up.

## Decision

v0.5.0 ships with UNOOSA + Celestrak only. Space-Track and full
skyfield propagation are deferred to v0.6.0 as an explicit, named
follow-up phase.

### Included in v0.5.0

- `src/orbital/ingest/celestrak.py` - public Celestrak GP client.
  No authentication, no rate-limit concerns beyond common courtesy.
- `src/orbital/transform/matching.py` - three-step hierarchical match
  (COSPAR, name+date, rapidfuzz) using Celestrak alone.
- `src/orbital/transform/orbit_regime.py` - **deterministic**
  classification of each Celestrak row into
  `Literal["LEO", "MEO", "GEO", "HEO", "unknown"]` from `mean_motion`
  and `eccentricity` fields already present in the GP catalog. No
  propagation, no ephemeris lookups, no skyfield dependency.
- `configs/canonical_schema.v1.yaml` + `CanonicalSchema` Pandera
  model per ADR-009.
- `pipelines/flows/canonical_flow.py` orchestrating the above.

### Deferred to v0.6.0

- `src/orbital/ingest/spacetrack.py` - authenticated Space-Track
  client. Credentials `SPACETRACK_USER` and `SPACETRACK_PASSWORD`
  are already provisioned in GitHub Actions secrets (reserved during
  v0.1.1 infra work); no work is lost by deferring.
- `src/orbital/transform/orbit_gap.py` - propagation-based analysis
  using historical TLEs to detect registered-but-untracked and
  tracked-but-unregistered asymmetries over time.
- `skyfield` as a runtime dependency.

### Dependency list for v0.5.0

Added: `rapidfuzz` only. `skyfield` and `spacetrack` remain in the
"Reserved for Fase 2/3" group in the stack documentation and are not
installed in v0.5.0 environments.

### What this does not change

- The canonical schema shape (ADR-009) is unaffected. The
  Space-Track-originating columns that ADR-009 anticipates do not
  exist - Space-Track does not introduce its own per-source columns
  in v0.5.0 because there is no Space-Track in v0.5.0.
  If v0.6.0 introduces per-source columns for Space-Track, they are
  added under the additive rule of ADR-008 (new nullable columns
  appended at the end).
- `orbit_regime_canonical` is populated in v0.5.0 via the
  deterministic classifier. `orbit_regime_confidence` is reserved per
  ADR-009 and remains null until a non-deterministic classifier
  lands, which is not scheduled for v0.6.0 either.
- The GitHub Actions secrets for Space-Track stay provisioned. No
  config or CI change is needed when v0.6.0 starts.

### Rough effort re-estimate

- v0.5.0 as scoped here: approximately 20 hours.
- v0.6.0 (Space-Track + orbit_gap + skyfield): approximately 15-20
  hours, as its own named phase with its own ADRs if the decisions
  there warrant them (Space-Track redistribution terms will almost
  certainly warrant one).

## Consequences

**Positive**

- v0.5.0 ships a publishable canonical dataset sooner, with a
  narrower surface area to review and release.
- No dependency on Space-Track's availability, rate limits, or
  terms of use for the v0.5.0 release.
- No unused `skyfield` dependency carried in v0.5.0 environments.
- v0.6.0 has a clear, bounded scope that can be reasoned about
  independently, including its own redistribution / licensing
  questions.

**Negative**

- The headline orbit_gap analysis that the dashboard thesis relies
  on does not ship in v0.5.0. The v0.5.0 dashboard narrative will
  need to focus on compliance asymmetry (registered vs tracked) at
  a single snapshot, not on temporal tracking gaps. This is a real
  narrative cost.
- Early consumers of the v0.5.0 canonical will see
  `orbit_regime_canonical` but no tracking-gap columns at all,
  which may prompt "is this dataset incomplete?" questions. The
  schema YAML and the Hugging Face / Kaggle dataset cards must
  state explicitly that Space-Track enrichment is a v0.6.0 feature.

**Neutral**

- The deterministic orbit_regime classifier may need a confidence
  companion in the future (ADR-009 reserves `orbit_regime_confidence`
  for exactly this). v0.5.0 ships it as null; v0.6.0 may or may not
  populate it depending on whether Space-Track data justifies a
  non-deterministic classifier.
- If v0.6.0 is never built (deprioritized, blocked on Space-Track
  terms, etc.), the v0.5.0 canonical is still internally coherent:
  `orbit_regime_canonical` is populated, per-source columns reflect
  only the sources that exist, and the schema carries no Space-Track
  nulls because no Space-Track columns were declared.
